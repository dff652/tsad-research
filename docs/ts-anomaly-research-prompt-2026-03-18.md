## 最终版提示词

请基于以下两个项目，启动一个“**工业传感器时序数据的异常检测算法研究与 Benchmark 建设**”任务：

- **项目1（现有业务平台）**：`/home/douff/ts/ts-iteration-loop`
- **项目2（算法研究环境）**：`/home/douff/my_project/autoresearch`

### 任务目标
使用项目2，对项目1中现有的时序异常检测算法进行：

1. **系统梳理**：区分基于模型与无模型方法，分析输入输出、参数、适用场景和优缺点
2. **统一 Benchmark 建设**：建立统一的数据、标签、指标和实验流程
3. **复现与对比测试**：在统一框架下复现或适配现有主要算法
4. **新算法调研与研发**：调研适合时序异常检测的新方法，并尝试研发或改进出优于现有方案的算法

### 重点要求
- 任务对象是 **时序异常检测**，不是泛化异常检测
- **数据现状与难点说明（必须考虑）**：
  - 真实业务数据（436个点位）**完全没有真实异常标签（Ground Truth）**，无法直接使用传统的 Precision/Recall/F1 等指标进行全自动评估。
  - 目前仅有 **100个采样点位** 的 **人工抽样评分数据**（5位专家对 Qwen-VL-8B、ChatTS、Timer 及 ADTK_HBOS 进行了打分和文字评价，0分/0.5分/1分）。
  - **为何采用人工评分**：由于工业时序场景复杂，不同工况下的异常定义主观性强，且缺乏专家标注资源，目前只能依赖这 100 个点位的主观评分作为 Benchmark 建设的唯一基准。
- **时序异常类型划分** 是重点，请先梳理并定义异常类型（如点异常、区间异常、趋势异常、变点/漂移、周期破坏、多变量关联异常等）
- **评估体系设计** 是重点，因为没有真实标签，请不仅考虑传统指标，更要基于现状设计：
  - **基于人工评分的量化指标**：如模型的平均得分、专家一致性（Fleiss' Kappa）、各维度分层得分等。
  - **无监督/弱监督评估指标**：如何利用 436 个全量无标签点位的推理特征（异常率、异常段分布、一致性等）来辅助判断算法稳定性。
  - 传统指标（如适用）：Precision, Recall, F1, False Positive Rate, Detection Delay
  - 稳定性 / 鲁棒性
  - 推理效率 / 成本（如适用）
- 最终目标是在统一评估框架下，找到或研发在关键指标上优于项目1当前方案的方法

### 算法运行环境解决方案（重要约束）
鉴于系统目前已经存在多个算法专属的 conda 环境（如 `chatts`, `timer`, `qwen_tune`, `ts-iteration-loop` 等），且 `autoresearch` 项目本身使用 `uv` 管理：
- **切勿尝试将所有算法合并到同一个环境**：深度学习模型依赖极易冲突（如不同版本的 PyTorch/CUDA）。
- **环境隔离与复用（针对现有算法）**：
  - `autoresearch` 的实验调度主干代码（如结果汇总、数据分发）可运行在其自带的 `uv` 环境或基础环境（如 `base` / `lab`）中。
  - 在涉及到具体某个大模型/算法推理或训练时，必须**复用其专属的 conda 环境**（例如通过 subprocess 执行 `conda run -n timer python xxx.py` 或由 Bash 脚本包装）。
  - 各算法的输入输出接口需在代码或脚本层面对齐（如统一从指定目录读取 CSV，将结果输出到另一个目录），以实现环境解耦。
- **环境搭建规范（针对新算法调研）**：
  - 每调研一个具有独立依赖体系的新算法时，**必须新建专属的 Conda 环境**（如 `conda create -n new_algo_name python=3.x`）。
  - 所有新算法的代码结构必须封装好独立的推理/训练脚本，且必须在“项目3 (`ts-anomaly-research`)”中为该算法提供环境配置文件（如 `requirements_[algo_name].txt` ），保证可复现。
  - 新算法同样通过进程调用的形式接入 Benchmark 调度端，保持与旧算法一致的“松耦合”接口。
- **其他研发与实验注意事项**：
  - **版本控制与阶段性提交**：本项目已连接到远程 GitHub 仓库，**每完成一个重大节点（如数据清洗完成、某项算法实验跑通出具报告、新增核心分析模块）**，必须执行一次 `git commit -m "feat/docs: ..."` 并 `git push` 给远程。
  - **敏感数据与源文件脱敏**：绝不可将真实工业 CSV 文件、打分 Excel 甚至原始脱敏映射表提交到 Git 仓库，必须严格维护 `.gitignore`，如果为了跑通流水线可以设立 `data/sample/` 并仅放入少量虚拟/高频加噪点位供测试。
  - **绝不可修改/破坏现有环境**：严格限制在各自沙盒中执行，无论是包安装还是版本降级。
  - **实验可追溯性**：所有实验（成功或失败）需在项目3留下包含 超参配置、模型权重（如适用）、推理预测结果CSV 及 评估指标的 完整存档（Log/Record）。
  - **异常处理与容错**：由于是自动化评估，请确保各个算法有良好的 Try-Except 机制，避免某个点位的推理崩溃导致整个 Benchmark 流程中断。

## 附件：初期研究设计与路线图指引

为指导后续实验的开展，特制定以下初步框架约定，后续自动执行时应以此作为参考骨架：

### 1. 时序异常类型划分（初步调研方向，待定论需持续完善）
*注：此分类为现有知识下整理的初始方向，非绝对定论，需在实际数据标注与算法分析过程中不断印证与增删。*
- **点异常**：全局离群（Spike/Dip）、上下文异常。
- **区间/片段异常**：均值漂移/阶跃、方差/波动变化、趋势形态改变。
- **模式异常**：周期性/工作流被破坏。
- **多变量异常**：传感器群相关物理量逻辑关联被破坏。

### 2. 项目1现有算法盘点框架（卡片登记规范）
每次评估或提取 `ts-iteration-loop` 中的现有算法时，必须至少厘清并输出：
1. 模型基本架构与超参数意义。
2. 输入特征要求与输出矩阵格式。
3. 所属专属 `conda` 环境名称及依赖硬性要求。
4. 推理/训练的算力开销和延迟瓶颈。
5. 优势场景与短板缺陷（例如对上述哪种类型的异常敏感/脱敏）。

### 3. Benchmark 分层评估体系设计
在彻底缺乏 Ground Truth 的窘境下，测试验证必须包含两个维度：
- **维度 A：人工先验核心集（100 个抽样评分点）拟合度**。计算预测结果与人工给定分数（0/0.5/1）的主观一致性或得分分布。
- **维度 B：无监督泛化集（436 个全量点）物理约束验证**。考核全量平均异常率是否符合工业常识（避免全面过检），并利用异常时序粘性、密度等特征硬性约束评估。

### 4. 重点新算法调研建议方向
不仅限于现有时序模型，鼓励重点尝试：
1. 具有零样本能力的时序大模型领域（TSFM）。
2. 不需要异常标签的自监督掩码重建网络结构。
3. 对噪声极具鲁棒性的扩散生成式检测框架。

### 5. Benchmark 建设 5 段式路线图
- **阶段 1（现状盘点与打底）**：理清现有异常算法存量，跑通并构建基于现成 `conda` 单独拉起的评测调度主循环代码。
- **阶段 2（指标量化实现）**：将文本化的人工主观经验和约束抽象化为数值评价代码函数。
- **阶段 3（沙盒接入基线）**：提取 2~3 个新算法在沙盒中建立依赖接入测试管道。
- **阶段 4（大规模拉锯战）**：在真实泛化集上反复让新老算法对撞、刷榜甚至设计定制化后处理阈值策略组件。
- **阶段 5（落地闭环）**：确定胜出者，整理一切可追溯的评估/验证 Log、重现脚本与迁移到业务平台的终本报告。
